<content_rules>

STEP 2 — INVENT NEW (only if Step 1 fails for all available material):

<bullet_point_rules>
- Begin every bullet with a strong action verb
- Include specific metrics, percentages, or numerical values in every bullet point
- Maximum 180 characters per \resumeItem{}
- Demonstrate leadership or process ownership in at least one bullet per section
- Limit repetition of any single word or action verb to 2 instances across the entire document
</bullet_point_rules>
<keyword_rules>
- Maintain 80% alignment with job description keywords for a natural, human-centric profile
- Bold all JD keywords chosen for inclusion: \textbf{Kubernetes}
- Filter JD keywords to extract only hard technical tools — discard conceptual methodologies or descriptive phrases
- Seamlessly integrate extracted keywords across both experience and project bullet points
- HARD COVERAGE RULE: Every framework or tool explicitly named in the JD (e.g., "CUDA, PyTorch, transformers, flash attention") must appear at least once somewhere in the resume — in a tech stack sidebar or a bullet point
- DOMAIN TARGETING: Identify the primary technical domain of the JD (e.g., LLM training, inference optimization, distributed ML) and ensure at least one bullet per relevant section references that domain directly using its specific terminology (e.g., inference latency, training throughput, KV cache, quantization, fine-tuning)
</keyword_rules>

</content_rules>

- Exactly 3 Experience sections and exactly 2 Project sections (5 total)

- Each skills category string must fit entirely on a single line
- Total content must fit precisely on one single page
- Properly escape all special LaTeX characters in all generated text: \& \% \$ \# \_
</formatting_constraints>
